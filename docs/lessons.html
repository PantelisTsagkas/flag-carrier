Flag Carrier

Why and how we built a daily European flag quiz for the office stand-up, and what it taught us about shipping small things properly.

STATUS DEPLOYED URL flag-carrier.fly.dev BUILT JUL 2026 COST ~PENNIES/MO

Why we built it

The office stand-up needed a ritual. The idea: a daily flag quiz where everyone gets the exact same ten flags, plays once, and the leaderboard settles it in person. The constraints came straight from that social contract:

  • The daily run must be identical for everyone, with no stored puzzle state to sync.
  • Your first submitted score stands. No grinding, no replays.
  • It has to be reachable by everyone with a link, cost near nothing, and need zero maintenance.

It is deliberately not a portfolio piece. That decision shaped everything: no framework frontend, no auth system, no Postgres. The engineering discipline went into the parts that matter (determinism, persistence, tests, cheap ops) and nowhere else.

How it works

BackendFastAPI + Uvicorn, one file (main.py), ~280 lines
FrontendSingle static index.html, vanilla JS, departures-board styling
StorageSQLite, one table, on a mounted volume
Packaginguv with a locked uv.lock, Docker image from the official uv base
HostingFly.io: one shared-cpu-1x / 256MB machine in LHR + 1GB volume
Testspytest + httpx against the FastAPI test client

The daily puzzle is a hash, not a database row

Every player gets the same rounds because the rounds are derived, not stored. The date string is hashed and the digest seeds Python's RNG, which then samples the flags:

def _seed(date_str: str) -> int:
    digest = hashlib.sha256(f"flag-carrier:{date_str}".encode()).digest()
    return int.from_bytes(digest[:8], "big")

rounds = _rounds_from_rng(random.Random(_seed(date_str)))

Same date in, same puzzle out, on any machine, with nothing to persist or invalidate. The day rolls over on the Europe/London date, so the puzzle changes at midnight UK time regardless of where the server runs.

Wrong answers are designed, not random

Random distractors make flag quizzes easy. The flags are grouped into look-alike families (Nordic crosses, Slavic tricolours, Monaco/Poland/Austria) and distractors are drawn from the answer's own family first. The traps people actually argue about at stand-up are guaranteed to appear.

The database enforces the game rule

"First score per name per day stands" is not application logic. It is a UNIQUE (date, name_key) constraint. A replay attempt raises IntegrityError, and the handler returns the original standing score instead. The rule cannot be bypassed by a race or a retry because the database is the referee.

How we deployed it

The app was Dockerized from day one, which made the hosting decision mechanical. The one real constraint was SQLite: it needs a disk that survives restarts, which rules out serverless (Vercel, Lambda) without a storage rewrite. Fly.io fit exactly: run the existing Dockerfile on a tiny VM, mount a volume, done.

# the entire deploy, after fly auth login
fly apps create flag-carrier
fly volumes create data --region lhr --size 1
fly deploy --ha=false

The cost-minimising choices in fly.toml:

  • auto_stop_machines = "stop" with min_machines_running = 0: the machine shuts down when idle and wakes on the next request. An office game is idle 95% of the time, so we pay for minutes, not months.
  • --ha=false: one machine, not two. Not just for cost: SQLite on a volume cannot be shared across machines, so high availability would actually corrupt the design.
  • Shared IPv4 instead of dedicated: saves $2/month and works fine behind Fly's proxy.
  • 256MB RAM: FastAPI serving JSON and static SVGs needs nothing more.

Lessons

1. The database choice is the architecture choice

Every hosting option was really a question about SQLite. Serverless platforms wipe local disk, so they were out unless we rewrote storage for Postgres. A file on a volume kept the app one file and the deploy one command. Pick the persistence model first; the platform follows.

2. Derive state instead of storing it

Seeding the daily puzzle from a hash of the date removed a whole class of work: no puzzle table, no generation job, no cache invalidation, no "what if two servers disagree". If a value can be recomputed deterministically, don't store it.

3. Put invariants in the database, not the code path

The one-score-per-day rule survives races, retries and curl because it is a UNIQUE constraint. Application-level checks ("query first, then insert") would have had a race window. Constraints are cheaper than correctness bugs.

4. Match the trust model to the audience, and say so out loud

Scores are computed client-side and posted to the API, so anyone with curl can submit a fake 2000. For an office game where cheaters get shamed in person, that is the correct amount of security. The lesson is not "client trust is fine", it is: know exactly what you are trusting, write it down, and revisit it if the audience changes.

5. Docker from day one makes deploy day boring

The image that ran on localhost is byte-for-byte the image running on Fly. Deploy day involved zero code changes: only a fly.toml, a volume, and an environment variable that already existed (FC_DB_PATH). Making the app configurable by env var before you need it is what made that possible.

6. Verify the deploy like you test the code

"It deployed" is not "it works". We smoke-tested every endpoint over the public URL, then restarted the machine and confirmed the leaderboard survived. That restart test proved the volume mount actually worked before anyone else touched the app. The scariest failure (scores silently vanishing on Monday) was ruled out in thirty seconds.

7. Cheap is a feature you design for

Under $1/month did not happen by accident. It came from auto-stop machines, a single instance, shared IPv4, and the smallest VM that works. Every one of those was a deliberate trade (cold starts, no HA) that is correct for this workload and would be wrong for another. Know which one you are running.

8. Scope discipline is what let it ship in a day

No accounts, no framework, no CSS library, no ORM. One Python file, one HTML file, one table. Everything cut was something that did not serve the stand-up. The corollary: the things we kept (tests, lockfile, Docker, README with a status line, honest trust-model notes) are the professional habits that cost little and pay off even in a toy.

If the audience ever grows

The known upgrade path, in order: server-side answer checking (the API hands out rounds without answers and validates guesses with timestamps), simple rate limiting or a shared passcode, and only then a hosted database if one machine stops being enough. None of it is needed while the URL stays inside the office.
